## Команды для разработки и тестирования

### Установка и настройка
```bash
uv sync              # Установить зависимости через менеджер пакетов uv
pre-commit run --all-files  # Запустить хуки линтинга/форматирования
```

### Запуск инструмента
```bash
# Базовое использование: организовать файлы из папки источника в целевую директорию
uv run media-sorter ~/source_folder ~/destination_folder

# С отображением прогресса и кастомным файлом логов
uv run media-sorter ~/Photos ~/Sorted --workers 4 --log migration.log

# Копировать файлы вместо перемещения их
uv run media-sorter ~/Source ~/Dest --copy

# Режим dry-run (симуляция без реальных операций)
uv run media-sorter ~/DCIM/Camera ~/Organized --dry-run
```

### Тестирование с pytest
```bash
# Запустить все тесты в проекте
pytest src/ -v

# Выполнение отдельного теста по шаблону
pytest <test_file.py>::<line_number>::<function_name> -v --tb=short

# Пример: запуск конкретной функции из модуля core
pytest src/core.py::test_extract_exif_date -v
```

### Линтинг и форматирование (через pre-commit)
- `pre-commit run ruff-check`  # Запустить линтер с исправлениями
- `pre-commit run ruff-format`  # Автоматическое форматирование кода
- `pre-commit run --all-files`  # Полный цикл линтинга/форматирования

## Обзор архитектуры проекта

### Структура проекта
```
media-sorter/
├── src/
│   ├── cli.py          # Точка входа CLI на Typer (команда main())
│   │                     - Проверяет директории источника и назначения
│   │                     - Отображает сводку конфигурации с иконками/цветами
│   │                     - Вызывает process_files() из модуля core
│   └── core.py         # Основная логика обработки:
│                         1. extract_exif_date()      - Чтение тега EXIF DateTime (Pillow)
│                         2. parse_date_from_filename()- Парсинг паттернов YYYYMMDD_* и кастомных форматов
│                         3. process_file()            - Создание структуры директорий, перемещение/копирование файлов
│                         4. process_files()             - Обработка параллельно через ThreadPoolExecutor (по умолчанию: 8 рабочих потоков)
├── src/logger.py       # Утилиты для логирования
│   ├── init_logger(log_file)     - Гарантирует существование родительских директорий, записывает шапку с временем
│   └── log_message(message, file)- Добавляет отформатированные сообщения со временными метками в файл логов
├── src/utils.py        # Вспомогательные функции для проверки путей:
│   ├── validate_directories()    - Проверяет существование источника, создаёт назначение при необходимости
│   ├── print_param(label,value)  - Красиво выводит параметры через Typer с иконками/цветами
│   └── validate_path(ctx,value)  # Callback, который вызывает BadParameter если путь не существует
├── src/config.py       # Константы: список DATE_FORMATS, EXIF_DATE_TAG="DateTime"
```

### Поток данных (Как это работает)
1. **Вход CLI** (`cli.py`): команда `main()` получает пути источника и назначения → проверяет валидность обеих директорий
2. **Сканирование файлов**: перебирает все файлы из папки источника (не подпапки, не символические ссылки на директории)
3. **Извлечение даты** (`core.py`): для каждого файла:
   - Сначала пытается `extract_exif_date()` через чтение EXIF с помощью Pillow → возвращает datetime если найдено
   - Переключается к `parse_date_from_filename()` для имен файлов по паттернам вроде `20230115_abc.jpg`, `photo-2024-07-20.png`
   - Пропускает файлы без обнаружимой даты (записывает "SKIPPED")
4. **Создание директорий**: создаёт вложенную структуру: `destination/YYYY/MM/` через mkdir(parents=True)
5. **Операция с файлом** (`process_file()`): 
   - Обработка параллельно через ThreadPoolExecutor(max_workers=workers, по умолчанию 8 рабочих потоков)
   - Использует `shutil.move()` по умолчанию или `shutil.copy()` если установлен флаг --copy
6. **Логирование**: все действия записываются в файл с форматом: `[timestamp] STATUS (source): filename -> year/month`

### Ключевые зависимости (`pyproject.toml`)
- Ядро приложения: typer, click, tqdm, pillow
- Инструменты разработки: pytest, black, ruff, mypy, python-dotenv, pre-commit

### Частые паттерны тестирования в этом проекте
1. **Тесты извлечения EXIF**: проверка `extract_exif_date()` корректно парсит теги DateTime через Pillow
2. **Тесты парсинга имени файла**: проверка `parse_date_from_filename()` обрабатывает различные форматы дат (YYYYMMDD, YYYY-MM-DD и т.д.)
3. **Тесты обработки файлов**: валидация `process_file()` создаёт правильную структуру директорий и перемещает/копирует файлы

## Решение распространённых проблем

### Ошибка "No files found" (Файлов не найдено)
- Убедитесь, что имена файлов содержат даты по поддерживаемым паттернам: `20230115_abc.jpg`, `photo-2024-07-20.png`
- Проверьте права доступа к папке источника через `ls -la ~/source_folder/`
- Убедитесь, что флаг --dry-run показывает какие файлы будут обработаны

### Ошибки разрешений на Linux/Mac
```bash
chmod +x src/cli.py  # Сделать скрипт исполняемым при необходимости
# Или убедиться, что назначение записываемое: mkdir -p ~/dest && chmod 755 ~/dest
```

### Данные EXIF не извлечены
- Некоторые форматы изображений не хранят данные EXIF (например, определённые PNG, JPEG вариации)
- Инструмент автоматически переключается на парсинг имени файла если дата не найдена в EXIF
- Проверьте логи для сообщений "SKIPPED: No date found"

### Проблемы с параллельной обработкой
- Рабочие потоки по умолчанию: 8 (настраиваемый через флаг --workers, диапазон: 1-32)
- Если файлы заблокированы другим процессом, уменьшите количество рабочих или используйте dry-run сначала
- Режим dry-run помогает проверить план перед реальными операциями с файлами

## Рабочий процесс разработки для Claude Code

### Типичный рабочий поток
```bash
# 1. Установить зависимости (один раз на окружение)
uv sync

# 2. Запустить линтинг/форматирование автоисправлений
pre-commit run --all-files

# 3. Проверить изменения локально через тесты
pytest src/core.py -v

# 4. Dry-run инструмента для проверки поведения перед реальными перемещениями
uv run media-sorter ~/source ~/dest --dry-run --workers 8

# 5. Проверить логи на наличие проблем или пропущенных файлов
cat log.txt | grep "SKIPPED\|ERROR" || echo "Ошибок не найдено!"

# 6. Запустить инструмент (или режим copy для сохранения оригиналов)
uv run media-sorter ~/source ~/dest --copy
```

### Паттерны изменения кода
- **Изменения CLI**: редактировать `src/cli.py` → добавить новые опции Typer в сигнатуру main() функции
- **Логика обработки**: править `src/core.py` → изменить функции извлечения даты или работы с файлами  
- **Конфигурация логирования**: обновлять `src/logger.py` → изменить формат логов в init_logger/log_message

### Правила проверки файлов (utils.py)
- Директория источника должна существовать (`validate_path()` callback проверяет это через параметр exists=True Typer)
- Назначение может создаваться автоматически самим инструментом
- Оба пути проверяются перед началом обработки для предотвращения частичных сбоев

## Специфичные архитектурные заметки для Claude Code

### Паттерн параллельной обработки (ThreadPoolExecutor)
```python
# Из core.py: process_files() использует ThreadPoolExecutor с max_workers=workers
with tqdm(total=len(files), desc="Обработка файлов") as pbar:
    with ThreadPoolExecutor(max_workers=workers):  # по умолчанию: 8 рабочих потоков
        futures = {executor.submit(process_file, ...): filename for filename in files}

        for future in as_completed(futures):
            future.result()  # Ожидание завершения и обработка исключений
            pbar.update(1)
```
- Файлы обрабатываются одновременно (не последовательно)
- Каждая операция с файлом независимая → безопасно для параллелизации
- Исключения от отдельных файлов не останавливают весь пакет

### Приоритет обнаружения даты (`core.py`)
1. Извлечение тега EXIF DateTime через Pillow's `img.getexif()` 
2. Парсинг имени файла по множеству форматов: `%Y%m%d%H%M%S`, `%Y-%m-%d %H.%M.%S` и т.д. (из config.py)

### Паттерн обработки ошибок
- Отдельные сбои файлов записываются в логи но не останавливают весь пакет
- Возвращает `False` на ошибку, `None` если пропущен, `True` при успехе по файлу
- На уровне пакета: все действия записываются в log_file с префиксом `[timestamp] STATUS ...`
